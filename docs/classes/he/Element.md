# תיאור המחלקה Element

## 1. מידע כללי
*   **שם המחלקה:** `Element`
*   **סוג:** מחלקה רגילה (POJO / Model)
*   **תפקיד:** מחלקה זו משמשת כתבנית או "מיכל" עבור יסוד כימי יחיד (כמו מימן או חמצן). היא מאחסנת את כל המאפיינים של היסוד.
*   **אינטראקציה:** 
    *   `MainActivity` יוצרת רשימה של אובייקטים אלו לאחר ניתוח ה-JSON.
    *   `ElementAdapter` משתמש באובייקטים אלו כדי למלא את שדות הטקסט בממשק המשתמש.

## 2. משתנים (שדות המחלקה)

| שם | סוג | מטרה |
| :--- | :--- | :--- |
| `name` | `String` | השם המלא של היסוד (למשל, "Helium"). |
| `symbol` | `String` | הסמל הקצר (למשل, "He"). |
| `number` | `int` | המספר האטומי. |
| `atomic_mass` | `double` | המסה האטומית. |
| `summary` | `String` | תיאור קצר או עובדות על היסוד. |

*כיצד זה בשימוש:* המשתנים הללו הם פרטיים (private). הגישה אליהם נעשית דרך שיטות "Getters" ו-"Setters" כדי להבטיח את בטיחות הנתונים.

## 3. מתודות המחלקה
המחלקה מורכבת בעיקר מ-**Getters** ו-**Setters**.

### שם המתודה: `getName` / `setName`
*   **סוג:** `public`
*   **ערך חזרה:** `String` (עבור get) / `void` (עבור set)
*   **מה המתודה עושה:** `getName` מחזירה את השם, `setName` מעדכנת אותו.

## 4. לוגיקה כללית
זהו "נושא נתונים". אתם יכולים לחשוב על זה כעל כרטיסייה בקטלוג. כל כרטיסייה מייצגת יסוד אחד ויש לה שדות לשם שלו, למשקל ולסמל.

## 5. הסבר במילים פשוטות
חשבו על המחלקה הזו כעל **כרטיס ביקור** של יסוד כימי.
על הכרטיס יש שדות ספציפיים: "שם", "סמל", "מספר טלפון" (מספר אטומי).
המחלקה עצמה היא רק ה*עיצוב* של הכרטיס. כשהאפליקציה רצה, היא יוצרת 118 כרטיסים "פיזיים" (אובייקטים) וממלאת אותם במידע.
